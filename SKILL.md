# 吊椅的笔记

把多来源素材整理成可直接保存到备忘录、Markdown 或发布草稿的中文笔记。目标是信息密、事实稳、读起来像熟悉技术圈的人在做判断，不像营销稿、新闻稿或模板化 AI 文章。

## 工作流程

1. 先判断来源类型：GitHub、公众号/博客、X/Twitter、截图/图片文字、论文/文档、个人草稿，或混合素材。
2. 收集来源事实：标题、链接、作者/机构、时间、数字、命令、项目名、论文名、功能点、原始论点。
3. 外部来源不足时查原始材料。GitHub 项目必须读 README 或文档；新闻、论文、工具更新和 API/CLI 用法必须以当前可访问来源为准。
4. 如果素材来自图片或截图，先转写可读文字；看不清的内容标注“不确定”，不要补写。
5. 根据来源类型读取 `references/writing-formats.md` 里的对应格式。
6. 输出干净 Markdown。默认用段落推进，少用列表；只有操作步骤、流程、对比项清楚时才用列表。

## 事实约束

- 不写素材里没有支撑的事实、数字、命令、协议、版本、时间线或人物表态。
- 重要链接必须保留；多个来源可以集中放在开头或结尾，按用户用途调整。
- 项目、产品、论文和工具名保留英文原名；中文解释放在后面。
- README、论文、公告、网页正文里有明确命令或配置时，可以引用；没有看到就不要写。
- 用户要“最新”“最近”“今天”等内容时，查当前来源再写。
- 不能验证的内容要明说，例如“素材里没有看到许可证信息”。

## 默认结构

除非用户指定格式，按这个顺序写：

1. 标题：对象名 + 观点、定位或问题。
2. 来源：保留 GitHub、公众号、X、论文或网页链接。
3. 第一段：说明这是什么，以及为什么值得记。
4. 中间段：展开背景、机制、数据、场景、功能、争议或启发。
5. 用法或实践：有命令、部署、配置、工作流、案例时写进去。
6. 结尾：给出定位判断，避免空泛口号。

## 风格要求

- 中文自然，像同事写给自己和同行看的笔记。
- 句子有节奏，但不要装腔；观点可以锋利，事实要克制。
- 多用具体对象、具体场景、具体数字；少用抽象形容词。
- 避免模板句、过度总结、营销词和夸张词。
- 不把内容写成短列表合集；段落之间要有推进关系。
- 默认 1000-3000 字；短素材可以 400-900 字；深度文章可以更长。

## 来源类型

- GitHub 仓库或 Gist：按 GitHub 版写，必须读 README、仓库说明或 Gist 内容。
- 公众号、博客、长网页：按公众号版写，保留原文链接，重组为更清晰的中文笔记。
- X/Twitter 帖子：按 X 推文版写，把短观点扩展成背景、判断、影响和实践意义。
- 截图、图片、PPT 或 OCR：按图片文字版写，先转写，再整理。
- 混合素材：按混合素材版写，以信息最完整的来源为主，其余来源作为证据或语境。
