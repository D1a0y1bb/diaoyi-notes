# 写作格式参考

这些规则来自用户公开可用任务与本机备忘录文章样本的格式提炼，不包含原始备忘录正文。

## 通用笔记格式

标题通常直接给出对象和判断，例如“项目名：它解决了什么问题”或“事件名：从现象到趋势”。正文开头不要铺垫太久，第一段就说明素材是什么、它为什么值得记录。后面用背景、机制、数据、案例、用法或影响逐层展开。

文章整体以自然段为主。列表只用于操作步骤、工具链、对比项、流程节点。不要把一篇文章写成知识点清单。常见长度是 1000-3000 字；短素材可以更短，但也要保留判断和上下文。

保留来源链接。链接可以放在标题后，也可以放在第一段之前。多个链接不要打散到每段里，除非链接本身是文章讨论对象。

## 公众号版

适用：微信公众号文章、博客长文、访谈整理、技术团队文章、行业观察。

版式：

```markdown
标题

原文来自：https://...

正文第一段...
```

写法：

- 第一段说明文章讲的核心问题，以及为什么值得重新整理。
- 第二段补背景，把读者带到真实场景里。
- 中间段展开文章里的关键事实、数据、方法、争议和案例。
- 可以加入用户视角的判断，但必须能从素材推出。
- 如果原文很长，把逻辑重排得更清楚，不复述目录。
- 图片或截图位置可以用“图中信息显示……”处理，不需要保留图片占位。

语气：

- 像技术圈观察笔记，不像媒体新闻。
- 可以有判断，但不要夸张。
- 少用小标题。素材本身结构很强时，可以保留 2-5 个短标题。

## GitHub 版

适用：GitHub 仓库、Gist、开源项目、Skill、CLI 工具、框架、模板项目。

版式：

```markdown
项目名：一句定位

GitHub：https://github.com/owner/repo

正文第一段...
```

必须做：

- 读取 README、仓库说明、文档或 Gist 原文。
- 保留真实仓库地址。
- 写清楚项目解决什么问题、为什么有价值、怎么工作、怎么用。
- 保留安装命令、部署方式、配置方式、环境变量、许可证、语言、技术栈等可见事实。
- README 没写的技术细节不要补。

结构：

- 第一段：项目是什么，核心承诺是什么。
- 第二段：它要解决的真实问题。
- 中间段：功能、架构、输入输出、工作流、适用场景。
- 用法段：安装、命令、部署、调用方式。
- 结尾：项目定位和客观信息。

写法：

- 技术名词保留英文原词，中文解释跟在旁边。
- 可以写“这个项目真正有意思的地方在于……”，但后面必须接具体机制。
- 不要写成 README 翻译；要写成读完源码入口或文档后的介绍笔记。

## X 推文版

适用：X/Twitter 帖子、短观点、截图转发、社交媒体讨论。

版式：

```markdown
标题

来源：https://x.com/...

正文第一段...
```

写法：

- 不把推文逐句翻译成中文。
- 先说明这条推文触及的问题，再写为什么它会引发讨论。
- 如果推文只有一句观点，要补齐背景、行业语境、可能影响和实践意义。
- 保留可验证数字，例如点赞、浏览、发布时间、项目 star、论文数量；看不到就不写。
- 可以把一个短观点扩展成完整笔记，但不能把作者没说的话写成作者观点。

常见结构：

- 现象：这条推文说了什么。
- 背景：为什么现在讨论它。
- 判断：它揭示了什么变化。
- 影响：对开发者、团队、产品或安全实践意味着什么。
- 结尾：给出可记住的一句话定位。

## 图片文字版

适用：截图、图片文字、PPT、长图、抖音图文、聊天截图、OCR 文本。

处理顺序：

1. 转写图片中可读文字。
2. 区分原文、推测、上下文补充。
3. 看不清的地方写“不确定”或略过。
4. 再按内容类型选择公众号版、GitHub 版或 X 推文版。

写法：

- 不保留无意义的 OCR 噪声。
- 图片里只有金句时，扩展为观点笔记；图片里有流程图时，写出流程关系。
- PPT 或架构图要先说明图在表达什么，再解释节点关系。
- 不能因为图片里出现项目名就假设项目功能，必须有文字或外部来源支撑。

## 避免的写法

- 不写“史上最强”“震撼发布”“颠覆行业”这类营销词。
- 不写泛泛的结论，例如“这对行业有重要意义”，除非后面有具体解释。
- 不把所有段落都写成“首先、其次、最后”。
- 不输出看起来工整但没有信息量的段落。
- 不用素材外的想象填补事实空白。
